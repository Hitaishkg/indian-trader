## 1. Purpose

`src/config/settings.py` is the single source of truth for all configuration in the Indian Trader system. It loads environment variables from `.env` at import time, validates that every required variable is present and well-formed, coerces string values into typed Python fields, and exposes a frozen `Settings` dataclass as a module-level singleton. Every other module in the system imports `settings` from this module instead of reading `os.environ` directly. If any required variable is missing or invalid, the module raises `ConfigurationError` at startup with a message listing all problems at once — the system never starts in a half-configured state. Secret values are never logged, printed, or included in string representations.

## 4. Settings Dataclass

A frozen dataclass with typed fields for every configuration value. Fields are grouped by category. The class must not have a `__repr__` or `__str__` that exposes secret values — override both to mask secrets.

```python
@dataclass(frozen=True)
class Settings:
    """Immutable configuration loaded from .env at startup.

    All secret fields are masked in __repr__ and __str__.
    """

    # --- Trading controls ---
    live_trading: bool
    paper_trading: bool
    max_trade_amount: int

    # --- Database ---
    database_url: str

    # --- Logging ---
    log_level: str

    # --- Broker credentials (Shoonya) ---
    shoonya_user: str
    shoonya_password: str
    shoonya_totp_secret: str

    # --- API keys ---
    fyers_api_key: str
    groq_api_key: str
    gemini_api_key: str
    github_pat: str
    brave_api_key: str

    # --- Notification ---
    telegram_bot_token: str
    telegram_chat_id: str
    gmail_credentials: str
```

### Custom `__repr__` and `__str__`

Both must replace the values of all secret fields (see Section 11) with the literal string `"***"`. Non-secret fields are displayed normally. Example output:

```
Settings(live_trading=False, paper_trading=True, max_trade_amount=10000, database_url='sqlite:///data/trading.db', log_level='INFO', shoonya_user='***', shoonya_password='***', shoonya_totp_secret='***', fyers_api_key='***', groq_api_key='***', gemini_api_key='***', github_pat='***', brave_api_key='***', telegram_bot_token='***', telegram_chat_id='***', gmail_credentials='***')
```

The `__repr__` must be explicitly defined (do not rely on the default frozen dataclass repr). The `__str__` must return the same output as `__repr__`.

## 6. `load_settings()` Public Function

```python
def load_settings(env_path: str | None = None) -> Settings:
    """Load and validate all configuration from .env, returning a Settings instance.

    Reads the .env file using python-dotenv, validates every required variable
    is present, validates value constraints, coerces types, and returns a
    frozen Settings dataclass.

    Args:
        env_path: Optional path to .env file. When None, python-dotenv
                  searches from the current working directory upward.
                  Useful for testing with alternate .env files.

    Returns:
        A fully validated, immutable Settings instance.

    Raises:
        ConfigurationError: If any required variable is missing, or if any
                            variable fails its validation rule. The error
                            message lists ALL problems found, not just the
                            first one.
    """
```

### Implementation contract

1. Call `dotenv.load_dotenv(dotenv_path=env_path, override=True)` to load `.env` into `os.environ`.
2. Collect all errors into a `list[str]`. Do not raise on the first error.
3. For each required variable: check `os.environ.get(VAR_NAME)`. If missing or empty string, append `"Missing required variable: VAR_NAME"` to the error list.
4. For each present variable: apply the validation rule from Section 7. If invalid, append the specific error message to the error list.
5. If the error list is non-empty after checking all variables, raise `ConfigurationError` with all errors.
6. If all checks pass, construct and return the `Settings` instance with coerced values.

## 8. Startup Failure Format

### `ConfigurationError` exception

```python
class ConfigurationError(Exception):
    """Raised when environment configuration is missing or invalid.

    Attributes:
        errors: List of all validation error messages found.
    """

    def __init__(self, errors: list[str]) -> None:
        self.errors = errors
        joined = "\n  - ".join(errors)
        super().__init__(
            f"Configuration validation failed with {len(errors)} error(s):\n  - {joined}"
        )
```

This guarantees:
- ALL missing/invalid variables are reported in a single exception.
- The user does not have to fix one variable, re-run, discover the next, repeat.
- The `errors` attribute is programmatically accessible for testing.

Example output when 3 variables are missing:

```
ConfigurationError: Configuration validation failed with 3 error(s):
  - Missing required variable: SHOONYA_USER
  - Missing required variable: SHOONYA_PASSWORD
  - MAX_TRADE_AMOUNT must be a positive integer <= 10000, got: '50000'
```

## 9. Type Coercion

String environment variable values are coerced to typed Python fields as follows:

| Target type | Coercion rule |
|---|---|
| `bool` | `value.strip().lower() == "true"` yields `True`; `"false"` yields `False`. Any other string is a validation error (not coerced to `False`). |
| `int` | `int(value.strip())`. Must succeed and result must satisfy the variable's specific constraints (e.g., positive, <= 10000). `ValueError` from `int()` is caught and reported as a validation error. |
| `str` | `value.strip()`. Whitespace is stripped from both ends. The result must be non-empty for required variables. |

No `float` or `Path` coercions are needed in this module. `DATABASE_URL` is stored as `str` — the caller extracts the path portion when opening a connection.

## 10. Module-Level Singleton

After the `load_settings()` function definition, the module must expose a singleton:

```python
settings: Settings = load_settings()
```

This line executes at import time. The first module to `from src.config.settings import settings` triggers `.env` loading and validation. If validation fails, `ConfigurationError` propagates immediately and the process cannot start.

Other modules consume settings like this:

```python
from src.config.settings import settings

db_path = settings.database_url.replace("sqlite:///", "")
```

### Testing consideration

The module-level `settings = load_settings()` will fail if tests import the module without a valid `.env`. To handle this, the Coder Agent must structure the module so that:

1. `load_settings(env_path=...)` is independently callable and testable.
2. The module-level singleton assignment is at the bottom of the file.
3. Tests that need to test `load_settings()` directly should call it with an explicit `env_path` pointing to a test `.env` file, rather than importing the `settings` singleton.
4. Tests that need to import `settings` should set up environment variables before the import, or use `importlib.reload`.

## 11. Security Rules

The following fields are classified as **secret**. They must NEVER appear in logs, print statements, error messages, `__repr__`, `__str__`, or any other string output:

| Field | Env variable |
|---|---|
| `shoonya_user` | `SHOONYA_USER` |
| `shoonya_password` | `SHOONYA_PASSWORD` |
| `shoonya_totp_secret` | `SHOONYA_TOTP_SECRET` |
| `fyers_api_key` | `FYERS_API_KEY` |
| `groq_api_key` | `GROQ_API_KEY` |
| `gemini_api_key` | `GEMINI_API_KEY` |
| `github_pat` | `GITHUB_PAT` |
| `brave_api_key` | `BRAVE_API_KEY` |
| `telegram_bot_token` | `TELEGRAM_BOT_TOKEN` |
| `telegram_chat_id` | `TELEGRAM_CHAT_ID` |
| `gmail_credentials` | `GMAIL_CREDENTIALS` |

Implementation: define a module-level constant `_SECRET_FIELDS: frozenset[str]` containing the field names. The custom `__repr__` and `__str__` iterate over dataclass fields and mask any field whose name is in `_SECRET_FIELDS`.

Error messages for secret fields must never include the actual value. The "Missing required variable: VAR_NAME" format is safe because it only names the variable, not its value. Validation error messages for non-secret fields (like `MAX_TRADE_AMOUNT` and `LOG_LEVEL`) may include the invalid value for debugging.

## 13. Acceptance Criteria

The Tester Agent must verify all of the following. Each criterion must map to at least one test.

1. `load_settings()` returns a `Settings` instance when all required variables are present and valid in `.env`.
2. `ConfigurationError` is raised when a single required variable (`SHOONYA_USER`) is missing from `.env`.
3. `ConfigurationError` is raised listing ALL missing variables when multiple required variables are absent (not just the first one).
4. `ConfigurationError.errors` attribute contains the exact list of error strings.
5. `LIVE_TRADING` defaults to `False` when the variable is absent from `.env`.
6. `LOG_LEVEL` defaults to `"INFO"` when the variable is absent from `.env`.
7. `LIVE_TRADING="true"` (case-insensitive: `"True"`, `"TRUE"`) coerces to `bool True`.
8. `LIVE_TRADING="false"` coerces to `bool False`.
9. `LIVE_TRADING="yes"` raises `ConfigurationError` (not a valid boolean).
10. `MAX_TRADE_AMOUNT="10000"` coerces to `int 10000`.
11. `MAX_TRADE_AMOUNT="10001"` raises `ConfigurationError` (exceeds hard cap).
12. `MAX_TRADE_AMOUNT="0"` raises `ConfigurationError` (must be positive).
13. `MAX_TRADE_AMOUNT="abc"` raises `ConfigurationError` (not parseable as int).
14. `DATABASE_URL="sqlite:///data/trading.db"` passes validation.
15. `DATABASE_URL="postgres://localhost/db"` raises `ConfigurationError`.
16. `LOG_LEVEL="DEBUG"` passes and is stored as `"DEBUG"`.
17. `LOG_LEVEL="info"` passes and is stored as `"INFO"` (uppercased).
18. `LOG_LEVEL="TRACE"` raises `ConfigurationError`.
19. `Settings` is frozen — assigning to any field after construction raises `FrozenInstanceError`.
20. `repr(settings)` does not contain the actual value of any secret field — only `"***"`.
21. `str(settings)` does not contain the actual value of any secret field — only `"***"`.
22. `live_trading=True` and `paper_trading=True` simultaneously raises `ConfigurationError`.
23. `load_settings(env_path="/path/to/test.env")` reads from the specified file, not the default `.env`.
24. Empty string values for required variables (e.g., `SHOONYA_USER=""`) are treated as missing.
25. Whitespace-only values for required variables (e.g., `GROQ_API_KEY="   "`) are treated as missing.
26. `MAX_TRADE_AMOUNT="-5"` raises `ConfigurationError` (negative values rejected).
27. `mypy` passes with `--ignore-missing-imports` on this file with no errors.
28. `ruff check` passes with no errors on this file.

## 14. Explicit Non-Goals

This module does NOT:

- Open or interact with the SQLite database. It only stores the `DATABASE_URL` string. Other modules extract the path and open the connection themselves.
- Fetch data from any external source.
- Send notifications.
- Create directories or files (other than the `.env` loading side effect in `os.environ`).
- Validate that API keys or tokens are actually functional (e.g., by making a test API call). It only checks they are present and non-empty.
- Provide hot-reloading of configuration. Once `settings` is constructed at import time, it is immutable for the lifetime of the process.
- Read from any source other than `.env` and `os.environ`. No YAML, no JSON config files, no CLI arguments.
- Log anything to the database or to files. It only raises exceptions on failure.
- Validate `.env.example` or keep it in sync — that is a human responsibility.
