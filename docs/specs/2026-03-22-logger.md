## 1. Purpose

Structured logging module for the Indian Trader pipeline. Configures Python's
standard `logging` framework to emit log records to two destinations
simultaneously: the console (stderr) for developer visibility, and the SQLite
`agent_logs` table for persistent audit trails. Every agent action, data quality
score, and trade decision flows through this module. Existing modules that
already call `logging.getLogger(__name__)` (fetcher, cleaner, fundamentals)
continue to work without modification -- `setup_logging()` configures the root
logger so all child loggers inherit the SQLite handler automatically.

## 3. Public API

### 3.1. `setup_logging(db_path: str | None = None) -> None`

Configures the Python root logger with two handlers: a `StreamHandler` writing
to stderr and a `SQLiteHandler` writing to the `agent_logs` table.

**Args:**
- `db_path` -- absolute or project-relative path to the SQLite database file.
  When `None`, the path is derived from `settings.database_url` by stripping
  the `sqlite:///` prefix and resolving relative to the project root (see
  Section 6 for the exact algorithm). When a non-None string is passed, it is
  used as-is (absolute) or resolved relative to the project root (if relative).

**Returns:** `None`

**Raises:** Nothing. If the database path cannot be resolved or the database
cannot be opened, a warning is printed to stderr and only the StreamHandler is
attached. Logging infrastructure must never crash the pipeline.

**Behaviour:** See Section 6 for full detail.

---

### 3.2. `get_logger(name: str) -> logging.Logger`

Thin convenience wrapper around `logging.getLogger(name)`.

**Args:**
- `name` -- logger name, typically `__name__` of the calling module.

**Returns:** `logging.Logger` instance.

**Raises:** Nothing.

**Behaviour:** Returns `logging.getLogger(name)`. No additional configuration
is applied -- the returned logger inherits handlers from the root logger
(which `setup_logging()` has already configured). Existing modules may
optionally switch from `logging.getLogger(__name__)` to
`from src.utils.logger import get_logger; logger = get_logger(__name__)`,
but this is cosmetic -- both produce the same result.

---

### 3.3. `log_agent_action(agent_name: str, action: str, level: str = "INFO", symbol: str | None = None, result: str | None = None, data_quality_score: float | None = None) -> None`

Direct structured write to the `agent_logs` table, bypassing the standard
logging pipeline. Designed for trading agents in Phase 3/4 that need precise
control over every column in the agent_logs row.

**Args:**
- `agent_name` -- identifies the agent or module (e.g. `"screener_agent"`,
  `"risk_agent"`). Written to the `agent_name` column.
- `action` -- human-readable description of what happened. Written to the
  `action` column.
- `level` -- one of `"DEBUG"`, `"INFO"`, `"WARNING"`, `"ERROR"`, `"CRITICAL"`.
  Default `"INFO"`. Written to the `level` column. If the value is not one
  of these five strings, it is coerced to `"INFO"` and no error is raised.
- `symbol` -- NSE ticker symbol or `None`. Written to the `symbol` column.
- `result` -- one of `"ok"`, `"error"`, `"warning"`, `"skipped"`, or `None`.
  Written to the `result` column. No validation is performed on this value --
  callers may pass any short string.
- `data_quality_score` -- float between 0.0 and 1.0, or `None`. Written to
  the `data_quality_score` column.

**Returns:** `None`

**Raises:** Nothing. If the database write fails, the error is printed to
stderr. The function never raises.

**Behaviour:** See Section 7 for full detail.

## 4. SQLiteHandler class design

### Class: `SQLiteHandler(logging.Handler)`

A `logging.Handler` subclass that writes each `logging.LogRecord` to the
`agent_logs` SQLite table.

**Constructor: `__init__(self, db_path: str) -> None`**

- Stores `db_path` as `self._db_path: str`.
- Creates a `threading.Lock` as `self._lock: threading.Lock`.
- Sets `self._connection: sqlite3.Connection | None = None`.
  The connection is opened lazily on the first `emit()` call, not in the
  constructor. This prevents errors during handler construction from
  blocking `setup_logging()`.
- Calls `super().__init__()`.

**Method: `_ensure_connection(self) -> sqlite3.Connection | None`**

- If `self._connection` is not None, returns it.
- Otherwise, opens `sqlite3.connect(self._db_path)`.
- Applies WAL mode pragmas:
  ```sql
  PRAGMA journal_mode=WAL;
  PRAGMA busy_timeout=30000;
  PRAGMA cache_size=-64000;
  PRAGMA synchronous=NORMAL;
  ```
- Creates the `agent_logs` table (see Section 5).
- Stores the connection in `self._connection` and returns it.
- If any step fails, prints the error to stderr, sets
  `self._connection = None`, and returns `None`.

**Method: `emit(self, record: logging.LogRecord) -> None`**

- Acquires `self._lock`.
- Calls `self._ensure_connection()`. If it returns `None`, releases the lock
  and returns silently (no retry, no raise).
- Extracts fields from the `LogRecord`:
  - `logged_at`: current IST time as ISO 8601 string with timezone offset
    (e.g. `"2026-03-22T22:15:30+05:30"`). Uses `datetime.datetime.now(ZoneInfo("Asia/Kolkata")).isoformat(timespec="seconds")`.
    Does NOT use `record.created` -- uses wall-clock time at write, consistent
    with all other modules.
  - `agent_name`: `record.name` (the Python logger name, e.g.
    `"src.data.fetcher"`).
  - `level`: `record.levelname` (e.g. `"INFO"`, `"WARNING"`).
  - `action`: `self.format(record)` -- the fully formatted log message. If no
    formatter is set, falls back to `record.getMessage()`.
  - `symbol`: `getattr(record, "symbol", None)`. Callers pass this via
    `logger.info("msg", extra={"symbol": "INFY"})`.
  - `result`: `getattr(record, "result", None)`. Callers pass via
    `extra={"result": "ok"}`.
  - `data_quality_score`: `getattr(record, "data_quality_score", None)`.
    Callers pass via `extra={"data_quality_score": 0.85}`.
- Executes the INSERT (see Section 5 for the exact SQL).
- Calls `conn.commit()`.
- Releases `self._lock`.
- If any exception occurs during INSERT or commit, catches it, prints the
  error to stderr via `print(f"[logger] SQLiteHandler.emit failed: {exc}", file=sys.stderr)`,
  and returns. Never raises.

**Method: `write_row(self, logged_at: str, agent_name: str, level: str, action: str, symbol: str | None, result: str | None, data_quality_score: float | None) -> None`**

Public method for direct structured writes to `agent_logs`, bypassing the `LogRecord` pipeline. Used by `log_agent_action()` to avoid accessing private attributes.

- Acquires `self._lock`.
- Calls `self._ensure_connection()`. If it returns `None`, releases the lock and returns silently.
- Executes the INSERT with the provided parameters.
- Calls `conn.commit()`.
- Releases `self._lock`.
- If any exception occurs, prints to stderr via `print(f"[logger] SQLiteHandler.write_row failed: {exc}", file=sys.stderr)` and returns. Never raises.

---

**Method: `close(self) -> None`**

- Acquires `self._lock`.
- If `self._connection` is not None, calls `self._connection.close()` and
  sets `self._connection = None`.
- Calls `super().close()`.
- Releases `self._lock`.
- If the close fails, prints to stderr and swallows the exception.

**Thread safety:** Every call to `emit()` and `close()` acquires the
instance-level `threading.Lock`. All database operations happen inside the
lock. This makes the handler safe for use from multiple threads (e.g. the
Monitor Agent's 5-minute loop running concurrently with other agents).

## 5. SQLite schema

```sql
CREATE TABLE IF NOT EXISTS agent_logs (
    id                 INTEGER PRIMARY KEY AUTOINCREMENT,
    logged_at          TEXT    NOT NULL,
    agent_name         TEXT    NOT NULL,
    level              TEXT    NOT NULL,
    action             TEXT    NOT NULL,
    symbol             TEXT,
    result             TEXT,
    data_quality_score REAL
);
```

**INSERT statement used by SQLiteHandler.emit and log_agent_action:**

```sql
INSERT INTO agent_logs (logged_at, agent_name, level, action, symbol, result, data_quality_score)
VALUES (?, ?, ?, ?, ?, ?, ?);
```

**Schema migration note:** The existing `src/data/validator.py` creates an
`agent_logs` table with a different schema (columns: `agent_name`,
`event_type`, `symbol`, `detail`, `data_quality_score`, `timestamp_ist`).
This logger module defines the canonical schema going forward. Because we are
still in Phase 1 development with no production data, the database file
(`data/trading.db`) should be deleted before first run with logger.py
integrated, so the new schema takes effect. The Coder Agent must NOT modify
validator.py -- that is a separate task. The `CREATE TABLE IF NOT EXISTS`
statement means if an old-schema table already exists, the CREATE is a no-op
and the handler will fail on INSERT due to column mismatch. In that case, the
handler prints a stderr warning and degrades gracefully (console-only logging).
This is acceptable for development. A future task will migrate validator.py to
use the logger module's `log_agent_action()` function instead of its own
`_log_event()` helper.

## 6. setup_logging behaviour in detail

### 6.1. Database path resolution

1. If `db_path` is not None and is an absolute path (`os.path.isabs(db_path)` is True), use it as-is.
2. If `db_path` is not None and is a relative path, resolve it relative to the project root. The project root is defined as `os.path.dirname(os.path.dirname(os.path.dirname(os.path.abspath(__file__))))` (three levels up from `src/utils/logger.py`).
3. If `db_path` is None, read `settings.database_url`. Strip the `sqlite:///` prefix. If the remaining path is absolute, use it. If relative, resolve it relative to the project root (same as step 2).
4. Create the parent directory of the resolved path if it does not exist (`os.makedirs(parent, exist_ok=True)`).
5. If any step fails (e.g. `settings.database_url` does not start with `sqlite:///`), print a warning to stderr and set `resolved_db_path = None` (meaning: skip SQLite handler, console-only).

### 6.2. Idempotency

Before attaching handlers to the root logger, check whether a `SQLiteHandler`
is already present:

```python
root = logging.getLogger()
for handler in root.handlers:
    if isinstance(handler, SQLiteHandler):
        return  # already configured, do nothing
```

This makes `setup_logging()` safe to call multiple times. The second and
subsequent calls are no-ops.

### 6.3. Handler attachment

1. Get the root logger: `root = logging.getLogger()`.
2. Set the root logger's level to `getattr(logging, settings.log_level, logging.INFO)`.
3. Create a `logging.StreamHandler(sys.stderr)`.
4. Set the StreamHandler's level to `logging.DEBUG` (let the root logger's
   level do the filtering -- the handler accepts everything the root passes).
5. Create and attach the console formatter (see Section 8).
6. Add the StreamHandler to the root logger.
7. If `resolved_db_path` is not None (from step 6.1), create a
   `SQLiteHandler(resolved_db_path)` and add it to the root logger. Set the
   SQLiteHandler's level to `logging.DEBUG` (same reasoning as StreamHandler).
   Do NOT set a formatter on the SQLiteHandler -- it uses `record.getMessage()`
   directly for the `action` column, producing clean message text without
   timestamp or level prefix (those are stored in dedicated columns).
8. Remove any pre-existing `StreamHandler` instances from the root logger that
   were added by other modules' ad-hoc setup code (fetcher.py, cleaner.py,
   fundamentals.py all add a StreamHandler to root if none exists). Do this
   BEFORE adding the new handlers in steps 6-7. The removal logic:
   ```python
   root.handlers = [h for h in root.handlers if not isinstance(h, logging.StreamHandler)]
   ```
   This ensures no duplicate console output after `setup_logging()` runs.

### 6.4. Log level

The root logger's level is set to `getattr(logging, settings.log_level, logging.INFO)`.
If `settings.log_level` is not a valid Python logging level name, `logging.INFO`
is used as the fallback. No error is raised.

## 7. log_agent_action behaviour in detail

1. Compute `logged_at` as current IST ISO 8601 string:
   `datetime.datetime.now(ZoneInfo("Asia/Kolkata")).isoformat(timespec="seconds")`.
2. Validate `level`: if `level` is not in `{"DEBUG", "INFO", "WARNING", "ERROR", "CRITICAL"}`,
   coerce it to `"INFO"`.
3. Look for an active `SQLiteHandler` on the root logger:
   ```python
   handler = None
   for h in logging.getLogger().handlers:
       if isinstance(h, SQLiteHandler):
           handler = h
           break
   ```
4. If no `SQLiteHandler` is found, print a warning to stderr:
   `"[logger] log_agent_action called but no SQLiteHandler configured. Call setup_logging() first."`.
   Then return without writing. Do not raise.
5. If a `SQLiteHandler` is found, call `handler.write_row(logged_at, agent_name, level, action, symbol, result, data_quality_score)`. No direct access to private attributes.
6. If any exception occurs during the write, print the error to stderr and
   return. Never raise.

## 8. Console formatter format string

**Exact format string:**

```python
"[%(asctime)s] [%(levelname)s] %(name)s -- %(message)s"
```

**Date format for asctime:**

```python
"%Y-%m-%d %H:%M:%S"
```

**Timezone handling for console output:** Python's `logging.Formatter` uses
local system time for `%(asctime)s` by default. To force IST, the formatter
must override `formatTime`. Define a private class:

```python
class _ISTFormatter(logging.Formatter):
    """Formatter that forces IST (Asia/Kolkata) timestamps in console output."""

    _IST = ZoneInfo("Asia/Kolkata")

    def formatTime(self, record: logging.LogRecord, datefmt: str | None = None) -> str:
        dt = datetime.datetime.fromtimestamp(record.created, tz=self._IST)
        if datefmt:
            return dt.strftime(datefmt)
        return dt.strftime("%Y-%m-%d %H:%M:%S")
```

Attach this formatter to the StreamHandler:

```python
formatter = _ISTFormatter("[%(asctime)s] [%(levelname)s] %(name)s -- %(message)s", datefmt="%Y-%m-%d %H:%M:%S")
stream_handler.setFormatter(formatter)
```

**Example console output:**

```
[2026-03-22 22:15:30] [INFO] src.data.fetcher -- Cache hit for RELIANCE from yfinance
[2026-03-22 22:15:31] [WARNING] src.data.fundamentals -- Could not extract roe from Screener.in for ADANIENT
```

## 9. Error handling

### 9.1. Database write failure in SQLiteHandler.emit

If the INSERT or commit raises any exception (e.g. `sqlite3.OperationalError`,
`sqlite3.DatabaseError`):

1. Print to stderr: `"[logger] SQLiteHandler.emit failed: {exc}"`.
2. Do NOT close the connection or set it to None. The next `emit()` call will
   retry using the existing connection. If the database has a transient lock,
   this allows recovery.
3. Do NOT raise the exception. Return silently.
4. Do NOT attempt to reconnect. If the connection is permanently broken, every
   subsequent `emit()` will fail and print to stderr. This is correct -- the
   console handler continues to work, and the stderr warnings alert the
   developer.

### 9.2. Database path missing or unresolvable

If `setup_logging()` cannot resolve the database path:

1. Print to stderr: `"[logger] Could not resolve database path, SQLite logging disabled"`.
2. Attach only the StreamHandler to the root logger. The pipeline runs with
   console-only logging.

### 9.3. SQLiteHandler.emit with None connection

If `_ensure_connection()` returns `None` (database could not be opened):

1. The `emit()` method returns silently. No stderr output beyond what
   `_ensure_connection()` already printed.
2. The log record is lost for the SQLite destination but still appears on the
   console via the StreamHandler.

### 9.4. log_agent_action with no SQLiteHandler

If `log_agent_action()` is called before `setup_logging()` (or if
`setup_logging()` was called but SQLite handler was skipped):

1. Print to stderr: `"[logger] log_agent_action called but no SQLiteHandler configured. Call setup_logging() first."`.
2. Return without writing. Do not raise.

## 10. Backwards compatibility notes

**Existing modules need zero changes.** Here is why:

- `src/data/fetcher.py` (line 30): `logger = logging.getLogger(__name__)` --
  this creates a child logger named `"src.data.fetcher"`. When `setup_logging()`
  configures the root logger with the SQLiteHandler and StreamHandler, this
  child logger inherits both handlers automatically via Python's logger hierarchy.

- `src/data/cleaner.py` (line 28): Same pattern. Works identically.

- `src/data/fundamentals.py` (line 35): Same pattern. Works identically.

- All three modules also add a `StreamHandler` to `logging.root` if no handlers
  exist (the `if not logging.root.handlers` guard). This is harmless IF
  `setup_logging()` is called BEFORE any of these modules are imported. The
  `setup_logging()` function clears pre-existing StreamHandlers from the root
  logger (Section 6.3 step 8), so even if these modules' ad-hoc handlers were
  already attached, they are removed and replaced by the canonical one.

- All three modules call `logger.setLevel(getattr(logging, settings.log_level))`
  on their own logger instance. This is redundant when `setup_logging()` sets
  the root logger's level, but not harmful. The effective level is the more
  permissive of the two (root vs child), so if both are set to the same value
  (which they are -- both read `settings.log_level`), behaviour is identical.

**Call order requirement:** `main.py` must call `setup_logging()` as the very
first operation after importing settings, BEFORE importing any module from
`src.data.*`. This ensures the root logger has the SQLiteHandler attached before
any child loggers are created. If `main.py` imports `src.data.fetcher` before
calling `setup_logging()`, fetcher's module-level code will add its own
StreamHandler to root. `setup_logging()` will then clean it up when it runs,
but any log messages emitted during the import (before `setup_logging()`) will
go to console only, not to SQLite. This is acceptable but suboptimal.

**src/data/validator.py compatibility:** validator.py writes directly to
`agent_logs` using its own `_log_event()` helper with a different schema
(columns: `event_type`, `detail`, `timestamp_ist` instead of `level`, `action`,
`logged_at`). The logger module does NOT modify validator.py. The two schemas
are incompatible in the same table. In practice: if the database file is fresh
(deleted and recreated), logger.py creates the table with the canonical schema.
If the database already has the validator's schema, the `CREATE TABLE IF NOT EXISTS`
is a no-op and the handler's INSERT will fail (column mismatch), degrading
gracefully to console-only logging. A future task will migrate validator.py to
use `log_agent_action()`.

## 14. What this module does NOT do

- Does NOT send notifications. That is `src/utils/notifier.py` (Phase 1 step 7).
- Does NOT rotate log files. SQLite handles storage. Console output goes to
  stderr and is not persisted by this module.
- Does NOT create, read, or modify any table other than `agent_logs`.
- Does NOT import from any `src.*` module except `src.config.settings`.
- Does NOT modify `src/data/validator.py` or any other existing module.
- Does NOT filter or suppress log messages beyond the root logger's level
  setting. All filtering is level-based via Python's standard `logging` module.
- Does NOT provide file-based logging (no `FileHandler`). The two destinations
  are stderr (StreamHandler) and SQLite (SQLiteHandler). If file logging is
  needed in the future, it is a separate handler added by a separate module.
- Does NOT handle log message formatting for the SQLite handler. The `action`
  column stores the plain formatted message. Structured JSON logging is not
  used -- the structure comes from the dedicated columns (symbol, result,
  data_quality_score).
- Does NOT batch or buffer writes. Each log record is committed individually.
  At the volumes this system produces (hundreds of messages per day, not
  thousands per second), individual commits are acceptable and provide better
  crash resilience.
