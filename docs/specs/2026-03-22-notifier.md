## 1. Purpose

Dual-channel notification module for the Indian Trader pipeline. Sends
notifications via Telegram Bot API and Gmail API. Three notification types
exist: ALERT (both channels), CHECKPOINT (both channels), and INFO (Telegram
only). Every send attempt, success, failure, and skip is logged to the
`agent_logs` table via `log_agent_action()`. The module degrades gracefully
when credentials are not configured -- it logs the skip and continues without
raising. Importing the module triggers no network calls and no OAuth flow;
side effects occur only when a public send function is called.

## 3. Notification types

Define as a string enum:

```python
class NotificationType(Enum):
    """Notification severity/routing type."""

    ALERT = "ALERT"
    CHECKPOINT = "CHECKPOINT"
    INFO = "INFO"
```

Channel routing rules:

| Type | Telegram | Gmail |
|------|----------|-------|
| ALERT | Yes | Yes |
| CHECKPOINT | Yes | Yes |
| INFO | Yes | No |

## 4. Public API

### 4.1. `send_alert(subject: str, message: str) -> dict[str, bool]`

Sends an ALERT notification via both Telegram and Gmail.

**Args:**
- `subject` -- short subject line (e.g. `"Kill switch fired: drawdown > 15%"`).
  Used as Telegram message prefix and Gmail email subject.
- `message` -- full notification body. Plain text.

**Returns:** `{"telegram": bool, "gmail": bool}` -- True if that channel sent
successfully, False if it failed or was skipped.

**Behaviour:**
1. Calls `_send_telegram(NotificationType.ALERT, subject, message)`.
2. Calls `_send_gmail(NotificationType.ALERT, subject, message)`.
3. If both return False, calls `log_agent_action(agent_name="notifier",
   action="ALERT notification could not be sent: both channels failed or unconfigured",
   level="CRITICAL", result="error")`.
4. Returns the result dict.

**Raises:** Nothing. Never raises.

---

### 4.2. `send_checkpoint(subject: str, message: str) -> dict[str, bool]`

Sends a CHECKPOINT notification via both Telegram and Gmail.

**Args:**
- `subject` -- short subject line (e.g. `"Trade approval needed: HDFC Bank BUY"`).
- `message` -- full notification body with trade details.

**Returns:** `{"telegram": bool, "gmail": bool}`

**Behaviour:** Identical to `send_alert` except uses `NotificationType.CHECKPOINT`.
The CRITICAL log on dual failure uses `"CHECKPOINT notification could not be sent:
both channels failed or unconfigured"`.

**Raises:** Nothing. Never raises.

---

### 4.3. `send_info(message: str) -> dict[str, bool]`

Sends an INFO notification via Telegram only. Gmail is never attempted.

**Args:**
- `message` -- notification body. No separate subject line; the entire message
  is sent as the Telegram text.

**Returns:** `{"telegram": bool, "gmail": False}` -- gmail is always False
(not attempted, not logged as skipped).

**Behaviour:**
1. Calls `_send_telegram(NotificationType.INFO, "", message)`. The subject
   parameter is empty string for INFO type; `_send_telegram` handles this by
   sending the message without a subject prefix.
2. Sets `gmail_result = False`. No Gmail call. No log entry for Gmail skip
   on INFO type.
3. Returns the result dict.

**Raises:** Nothing. Never raises.

## 5. Private functions

### 5.1. `_send_telegram(notification_type: NotificationType, subject: str, message: str) -> bool`

Sends a single message via the Telegram Bot API.

**Args:**
- `notification_type` -- used in the log action string.
- `subject` -- prepended to the message with a newline if non-empty.
- `message` -- the body text.

**Returns:** True if the HTTP POST returned status 200 and the response JSON
has `"ok": true`. False otherwise.

**Behaviour:**
1. Check `settings.telegram_bot_token` and `settings.telegram_chat_id`. If
   either is None (or empty string after strip): log the skip and return False.
   See Section 9 for the exact log call.
2. Construct the Telegram text. If `subject` is non-empty:
   `text = f"<b>[{notification_type.value}] {subject}</b>\n\n{message}"`.
   If `subject` is empty (INFO type):
   `text = message`.
3. POST to the Telegram Bot API. See Section 7 for full detail.
4. On success: log via `log_agent_action()` and return True.
5. On failure: log via `log_agent_action()` and return False.

---

### 5.2. `_send_gmail(notification_type: NotificationType, subject: str, message: str) -> bool`

Sends a single email via the Gmail API.

**Args:**
- `notification_type` -- used in email subject prefix and log action string.
- `subject` -- the subject line content.
- `message` -- the email body (plain text).

**Returns:** True if the Gmail API call succeeded. False otherwise.

**Behaviour:**
1. Check `settings.gmail_credentials`. If None (or empty string after strip):
   log the skip and return False. See Section 9 for the exact log call.
2. Call `_build_gmail_service()`. If it returns None: log the failure and
   return False.
3. Call `_build_mime_message(notification_type, subject, message)` to construct
   the MIME message.
4. Base64url-encode the MIME message and send via Gmail API. See Section 8 for
   full detail.
5. On success: log via `log_agent_action()` and return True.
6. On failure: log via `log_agent_action()` and return False.

---

### 5.3. `_build_gmail_service() -> Any | None`

Builds and returns a Gmail API service object, handling OAuth token management.

**Args:** None.

**Returns:** A `googleapiclient.discovery.Resource` object for the Gmail API,
or None if the service could not be built.

**Behaviour:** See Section 6 for the complete OAuth flow.

**Caching:** The function stores the built service in a module-level variable
`_gmail_service_cache: Any | None = None`. On subsequent calls, if the cache
is not None, it returns the cached service immediately without rebuilding.
This avoids re-reading token.json and re-authenticating on every send.

The cache is invalidated (set to None) only if a Gmail API call fails with an
authentication error, at which point `_send_gmail` sets `_gmail_service_cache`
to None so the next call rebuilds the service.

**Type annotation:** Return type is `Any | None` because
`googleapiclient.discovery.Resource` is not a concrete type that can be
imported for annotation. The lazy import makes a proper type annotation
impractical. This is acceptable.

---

### 5.4. `_build_mime_message(notification_type: NotificationType, subject: str, message: str) -> MIMEText`

Constructs a MIME email message.

**Args:**
- `notification_type` -- used in the subject prefix.
- `subject` -- the subject line content.
- `message` -- the email body (plain text).

**Returns:** A `MIMEText` object ready for base64url encoding.

**Behaviour:**
1. Create `MIMEText(message, "plain")`.
2. Set `msg["Subject"]` to `f"[Indian Trader] {notification_type.value}: {subject}"`.
3. Set `msg["From"]` to `"me"` (Gmail API resolves this to the authenticated
   user's address).
4. Set `msg["To"]` to `"me"` (Gmail API resolves this to the same address --
   the system emails itself).
5. Return the `MIMEText` object.

## 6. Gmail OAuth flow in detail

### 6.1. Scopes

```python
_GMAIL_SCOPES: list[str] = ["https://www.googleapis.com/auth/gmail.send"]
```

Defined as a module-level constant.

### 6.2. File paths

- **credentials.json**: The path is read from `settings.gmail_credentials`.
  This is the OAuth 2.0 client credentials file downloaded from Google Cloud
  Console. It is NOT stored in the project -- the user downloads it and sets
  the path in `.env`.

- **token.json**: Stored at `{PROJECT_ROOT}/token.json` where `PROJECT_ROOT`
  is `os.path.dirname(os.path.dirname(os.path.dirname(os.path.abspath(__file__))))`.
  This file is already in `.gitignore` (confirmed: line 30 of current
  `.gitignore`). Contains the OAuth refresh token and access token after
  first-run authorization.

### 6.3. `_build_gmail_service()` step-by-step

```
1. If _gmail_service_cache is not None, return it immediately.

2. Compute token_path = os.path.join(PROJECT_ROOT, "token.json").

3. creds: Credentials | None = None

4. If token.json exists at token_path:
   a. creds = Credentials.from_authorized_user_file(token_path, _GMAIL_SCOPES)

5. If creds is None OR creds.valid is False:
   a. If creds is not None AND creds.expired is True AND creds.refresh_token is not None:
      i.  Try: creds.refresh(GoogleAuthRequest())
      ii. On failure (any exception): log error, delete token.json, set creds = None.
          Fall through to step 5b.
   b. If creds is still None (no token.json, or refresh failed):
      i.   credentials_path = settings.gmail_credentials
      ii.  If credentials_path does not exist as a file: log error, return None.
      iii. flow = InstalledAppFlow.from_client_secrets_file(credentials_path, _GMAIL_SCOPES)
      iv.  creds = flow.run_local_server(port=0)
      v.   This opens a browser for the user to authorize. It is a blocking call.
           It runs ONCE -- on the very first Gmail send attempt when no token.json exists.
      vi.  On failure (any exception): log error, return None.

6. Save the credentials to token.json:
   a. with open(token_path, "w") as f: f.write(creds.to_json())
   b. On failure: log warning (token not persisted, next run will re-auth),
      but continue -- creds are still valid in memory.

7. service = build("gmail", "v1", credentials=creds)

8. Set _gmail_service_cache = service.

9. Return service.

10. If any unhandled exception occurs at any step: log error via
    log_agent_action(), return None. Never raise.
```

### 6.4. Token refresh behaviour

When `token.json` exists but the access token has expired:
- `creds.expired` is True and `creds.refresh_token` is not None.
- `creds.refresh(GoogleAuthRequest())` obtains a new access token using the
  refresh token. No browser interaction needed.
- The refreshed credentials are saved back to `token.json`.

When the refresh token itself is revoked or invalid:
- `creds.refresh()` raises an exception.
- The code deletes `token.json` and falls through to the full OAuth flow
  (step 5b), which opens the browser again.

### 6.5. First-run user experience

On the very first call to any `send_alert()` or `send_checkpoint()` function
(when `settings.gmail_credentials` is set but `token.json` does not exist):

1. The system opens a browser window for Google OAuth authorization.
2. The user authorizes the application.
3. The authorization code is exchanged for tokens.
4. `token.json` is saved to the project root.
5. All subsequent calls use the saved token (no browser needed).

This is a one-time operation. If the system is running headless (no browser),
the OAuth flow will fail and Gmail sends will return False. This is acceptable
-- Telegram still works independently.

## 7. Telegram API call in detail

### 7.1. Endpoint

```
https://api.telegram.org/bot{token}/sendMessage
```

Where `{token}` is `settings.telegram_bot_token`.

### 7.2. HTTP call

```python
response = requests.post(
    f"https://api.telegram.org/bot{settings.telegram_bot_token}/sendMessage",
    json={
        "chat_id": settings.telegram_chat_id,
        "text": text,
        "parse_mode": "HTML",
    },
    timeout=10,
)
```

### 7.3. Success determination

The call is considered successful if ALL of these are true:
- `response.status_code == 200`
- `response.json()` does not raise
- `response.json().get("ok") is True`

### 7.4. Error handling

Catch `requests.RequestException` (covers `ConnectionError`, `Timeout`,
`HTTPError`, and all other requests exceptions). On any exception:
- Log via `log_agent_action()`.
- Return False.
- Do NOT raise.

If the response status is not 200 or `"ok"` is not True:
- Log the status code and response text (truncated to 200 chars) via
  `log_agent_action()`.
- Return False.

### 7.5. Message length

Telegram's `sendMessage` has a 4096 character limit. If `text` exceeds 4096
characters, truncate to 4093 characters and append `"..."`. This is a
defensive measure -- in practice, notifications should be well under this
limit.

## 8. Gmail API call in detail

### 8.1. MIME construction

Handled by `_build_mime_message()` (Section 5.4).

### 8.2. Base64url encoding

```python
raw_bytes = mime_message.as_bytes()
encoded = base64.urlsafe_b64encode(raw_bytes).decode("ascii")
```

### 8.3. API call

```python
service.users().messages().send(
    userId="me",
    body={"raw": encoded},
).execute()
```

### 8.4. Success determination

The call is considered successful if `execute()` returns without raising an
exception. The return value (a dict with message ID) is not inspected.

### 8.5. Error handling

Catch `Exception` (the Gmail API can raise `googleapiclient.errors.HttpError`,
`google.auth.exceptions.RefreshError`, or other exceptions). On any exception:
- Log via `log_agent_action()` with the exception message.
- Set `_gmail_service_cache = None` (force service rebuild on next attempt).
- Return False.
- Do NOT raise.

## 9. Phase-gated degradation logic

### 9.1. Telegram credentials missing

Condition: `settings.telegram_bot_token is None` or
`settings.telegram_bot_token.strip() == ""` or
`settings.telegram_chat_id is None` or
`settings.telegram_chat_id.strip() == ""`.

Action in `_send_telegram`:
- Call `log_agent_action(agent_name="notifier", action="Telegram skipped: credentials not configured", level="WARNING", result="skipped")`.
- Return False.

Note: In the current `settings.py`, `telegram_bot_token` and
`telegram_chat_id` are always-required `str` fields (not `str | None`).
`settings.py` will raise `ConfigurationError` at startup if they are missing.
The None check in `_send_telegram` is defensive -- it protects notifier.py
against future changes to settings.py that might make these fields phase-gated.

### 9.2. Gmail credentials missing

Condition: `settings.gmail_credentials is None` or
`settings.gmail_credentials.strip() == ""`.

Action in `_send_gmail`:
- Call `log_agent_action(agent_name="notifier", action="Gmail skipped: credentials not configured", level="WARNING", result="skipped")`.
- Return False.

### 9.3. Both channels failed or skipped for ALERT or CHECKPOINT

Condition: Both `_send_telegram()` and `_send_gmail()` returned False for an
ALERT or CHECKPOINT notification.

Action in `send_alert` / `send_checkpoint`:
- Call `log_agent_action(agent_name="notifier", action="{TYPE} notification could not be sent: both channels failed or unconfigured", level="CRITICAL", result="error")`.
  Where `{TYPE}` is `"ALERT"` or `"CHECKPOINT"`.

### 9.4. INFO type -- Gmail skip is silent

For INFO notifications, Gmail is never attempted. No log entry is produced for
the Gmail skip. Only Telegram is attempted and logged.

## 16. What this module does NOT do

- Does NOT read from or write to any database table. All logging goes through
  `log_agent_action()` which handles the database write.
- Does NOT wait for or process replies from Telegram or email. It is
  fire-and-forget. Human reply handling is the Execution Agent's
  responsibility (Phase 4).
- Does NOT format trade-specific messages. Callers (agents) construct the
  message string and pass it in. Notifier handles delivery only.
- Does NOT retry failed sends. Each send is attempted once. If it fails,
  it is logged and returns False. Retry logic belongs to the calling agent
  if needed.
- Does NOT validate the content of messages (no profanity filter, no length
  validation beyond the 4096-char Telegram truncation).
- Does NOT manage Telegram bot creation or Gmail OAuth app registration.
  These are one-time manual setup tasks documented in the project README.
- Does NOT send SMS, push notifications, or any channel other than Telegram
  and Gmail.
- Does NOT import from any `src.*` module except `src.config.settings` and
  `src.utils.logger`.
- Does NOT store sent notification history. The `agent_logs` table entries
  created by `log_agent_action()` serve as the audit trail.
- Does NOT modify `settings.py`, `logger.py`, or any other existing module.
