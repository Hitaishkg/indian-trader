## 1. Purpose

`src/data/cleaner.py` sits between the fetcher and the validator in the data pipeline. It receives a raw OHLCV DataFrame from `src/data/fetcher.py` (already normalised to the Section 5.1 contract: columns `[symbol, date, open, high, low, close, volume]` with timezone-aware dates) and returns a cleaned DataFrame ready for the validator. The cleaner performs best-effort data repair: forward-filling missing prices, removing duplicate dates, and flagging anomalies such as negative prices and OHLCV inconsistencies. It never drops anomalous rows -- it flags them and leaves them in the DataFrame for the validator to assess. It never changes the schema, columns, dtypes, or sort order of the input. If no cleaning is needed, the DataFrame is returned unchanged. A `CleaningReport` is returned alongside the DataFrame to provide an audit trail of every repair and flag applied.

## 4. Input/Output Contract

### Input

A normalised OHLCV DataFrame matching the `validator.py` Section 5.1 contract, as produced by `fetcher.py`:

| Column | dtype | Description |
|---|---|---|
| `symbol` | `str` (object) | NSE ticker symbol, uppercase |
| `date` | `datetime64[ns, Asia/Kolkata]` | Trading date, timezone-aware IST |
| `open` | `float64` | Open price in INR |
| `high` | `float64` | High price in INR |
| `low` | `float64` | Low price in INR |
| `close` | `float64` | Close price in INR |
| `volume` | `float64` | Traded volume |

The DataFrame has a default integer index (RangeIndex) and is sorted by `(symbol, date)` ascending.

### Output

The same DataFrame with identical columns, dtypes, and sort order. Only values within existing columns may change (NaN filled, duplicate rows removed). The cleaner MUST NOT:

- Add columns
- Remove columns
- Change column dtypes
- Change the sort order (remains `(symbol, date)` ascending)
- Add rows

The cleaner MAY:

- Remove rows (only duplicate-date rows within a symbol group)
- Modify NaN values (forward-fill, zero-fill)

If no cleaning is needed, the input DataFrame is returned unchanged.

## 5. Cleaning Operations

Operations are applied in this exact order. The Coder Agent must implement them sequentially, not rearrange.

### 5a. Schema Validation

Before any cleaning, validate the input DataFrame structurally. This mirrors `_validate_ohlcv_df` in `validator.py`:

1. Check that all 7 required columns are present: `symbol`, `date`, `open`, `high`, `low`, `close`, `volume`.
2. Check that the `date` column is timezone-aware (has a non-None `tz` attribute).

If either check fails, raise `ValueError` immediately with a descriptive message listing the problem. Do not attempt any cleaning on a malformed DataFrame.

The Coder Agent should implement this as an internal function `_validate_schema(df: pd.DataFrame) -> None` that mirrors the logic of `_validate_ohlcv_df` in `validator.py`. Do NOT import `_validate_ohlcv_df` from validator -- it is a private function. Duplicate the logic to avoid coupling to a private API.

### 5b. Negative Price Detection

For each row, check whether any of `open`, `high`, `low`, `close` is less than or equal to zero.

- If so: record `(symbol, date_str)` in `CleaningReport.negative_price_flags`.
- Do NOT drop the row. Do NOT modify the values.
- Log each flagged row at WARNING level: `"Negative/zero price detected: {symbol} on {date_str}"`.

`date_str` is the date formatted as ISO 8601 date string (`YYYY-MM-DD`), extracted via `pd.Timestamp(date_value).strftime("%Y-%m-%d")`.

### 5c. OHLCV Consistency Check

For each row, check whether `high < low` (data corruption indicator).

- If so: record `(symbol, date_str)` in `CleaningReport.consistency_flags`.
- Do NOT drop the row. Do NOT modify the values.
- Log each flagged row at WARNING level: `"OHLCV inconsistency (high < low): {symbol} on {date_str}"`.

### 5d. Missing Value Handling

Handle NaN values within each symbol group independently. Forward-fill must NOT bleed across symbol boundaries.

Implementation order within this step:

1. **Group by symbol.** Use `df.groupby("symbol", group_keys=False)` to ensure operations are scoped per symbol.

2. **Forward-fill close price.** Within each symbol group, apply `ffill()` to the `close` column. Count the number of NaN values that were filled (compare `isna()` before and after). Record the total count in `CleaningReport.missing_close_filled`.

3. **Fill open/high/low from filled close.** After forward-filling close, for any row where `open`, `high`, or `low` is still NaN, set all three to the (now-filled) `close` value for that row. This handles rows where close was available (or was just forward-filled) but the other price columns were missing. Count all individual field fills (each of open, high, low counts separately) and add them to `CleaningReport.missing_ohlv_filled`.

4. **Fill volume with 0.** Replace any remaining NaN in `volume` with `0.0`. Count these fills and add to `CleaningReport.missing_ohlv_filled`.

Log at INFO level: `"Filled {n_close} close values, {n_ohlv} open/high/low/volume values across {n_symbols} symbols"`.

### 5e. Duplicate Date Removal

Within each symbol group, check for duplicate dates. If any exist:

1. Keep the **last** occurrence (the most recently added row wins).
2. Record the count of removed duplicates in `CleaningReport.duplicates_removed`.
3. Log each duplicate at WARNING level: `"Duplicate date removed: {symbol} on {date_str}"`.

Use `df.drop_duplicates(subset=["symbol", "date"], keep="last")` to implement this.

After deduplication, reset the index to maintain a clean RangeIndex.

### 5f. Price Floor Check

For each row in the (now cleaned) DataFrame, check whether `close < price_floor` (default 1.0).

- If so: record `(symbol, date_str)` in `CleaningReport.price_floor_flags`.
- Do NOT drop the row. Do NOT modify the values.
- Log each flagged row at WARNING level: `"Price below floor ({price_floor}): {symbol} on {date_str}, close={close}"`.

## 6. CleaningReport Dataclass

```python
@dataclass(frozen=True)
class CleaningReport:
    """Result of a cleaning pass over an OHLCV DataFrame.

    All flag lists contain (symbol, date_str) tuples where date_str
    is an ISO 8601 date string (YYYY-MM-DD).

    Attributes:
        symbols_processed: Symbols present in the input DataFrame.
        rows_input: Row count of the input DataFrame before cleaning.
        rows_output: Row count after cleaning (equals rows_input minus duplicates_removed).
        duplicates_removed: Count of rows removed due to duplicate dates within a symbol.
        missing_close_filled: Count of NaN close values that were forward-filled.
        missing_ohlv_filled: Count of open/high/low/volume NaN values filled.
        negative_price_flags: List of (symbol, date_str) for rows with any price <= 0.
        consistency_flags: List of (symbol, date_str) for rows where high < low.
        price_floor_flags: List of (symbol, date_str) for rows where close < price_floor.
        cleaned_at_ist: IST timestamp (ISO 8601 with timezone) when clean_ohlcv was called.
    """

    symbols_processed: list[str]
    rows_input: int
    rows_output: int
    duplicates_removed: int
    missing_close_filled: int
    missing_ohlv_filled: int
    negative_price_flags: list[tuple[str, str]]
    consistency_flags: list[tuple[str, str]]
    price_floor_flags: list[tuple[str, str]]
    cleaned_at_ist: str
```

All fields are required at construction -- no defaults. The dataclass is frozen (`frozen=True`) to prevent mutation after construction.

The `cleaned_at_ist` field is captured at the moment `clean_ohlcv` is called (at function entry, not at return). Format: `datetime.datetime.now(ZoneInfo("Asia/Kolkata")).isoformat(timespec="seconds")`.

## 7. Public API

Single public function:

```python
def clean_ohlcv(
    df: pd.DataFrame,
    price_floor: float = 1.0,
) -> tuple[pd.DataFrame, CleaningReport]:
    """Clean an OHLCV DataFrame by handling missing values, removing duplicates, and flagging anomalies.

    Receives a normalised OHLCV DataFrame (matching the validator.py Section 5.1
    contract) and returns a cleaned copy alongside a CleaningReport detailing
    every repair and flag applied. The cleaner never drops anomalous rows -- it
    flags them and leaves them for the validator to assess. If no cleaning is
    needed, the DataFrame is returned unchanged.

    Args:
        df: Normalised OHLCV DataFrame with columns [symbol, date, open, high,
            low, close, volume]. The date column must be timezone-aware
            (Asia/Kolkata). Produced by src.data.fetcher.fetch_ohlcv().
        price_floor: Minimum acceptable close price. Rows with close below
                     this value are flagged in the CleaningReport but not
                     removed. Default 1.0 (INR). This is a data sanity check,
                     not a strategy filter.

    Returns:
        A tuple of (cleaned_df, cleaning_report) where:
        - cleaned_df has the same schema as the input (same columns, same
          dtypes, same sort order). Only values may differ.
        - cleaning_report is a frozen CleaningReport with full audit trail.

    Raises:
        ValueError: If required columns are missing or dates are timezone-naive.
    """
```

The function operates on a **copy** of the input DataFrame. It must not mutate the caller's DataFrame. Use `df.copy()` at the start.

## 11. Error Handling

No bare `except` clauses. Every `except` block must catch a specific exception type.

| Situation | Exception type | Action |
|---|---|---|
| Required column missing from input DataFrame | Raise `ValueError` | Message lists missing columns |
| Date column is timezone-naive | Raise `ValueError` | Message states timezone requirement |
| Any other malformed input | Raise `ValueError` | Descriptive message |

The cleaner does NOT define a custom exception class. It is a best-effort fixer, not a gatekeeper. The validator (`DataQualityError`) is the gatekeeper. The cleaner raises `ValueError` only for structurally invalid input that cannot be cleaned at all.

The cleaner never halts on anomalies (negative prices, high < low, price floor violations). It flags them in the `CleaningReport` and returns normally. The caller decides what to do with the flags.

## 12. Acceptance Criteria

The Tester Agent must verify all of the following. Each criterion must map to at least one test.

### Schema validation

1. **Missing column raises ValueError.** Pass a DataFrame missing the `close` column. Verify `ValueError` is raised with a message listing the missing column.
2. **Multiple missing columns listed in error.** Pass a DataFrame missing both `open` and `volume`. Verify `ValueError` message lists both.
3. **Timezone-naive dates raise ValueError.** Pass a DataFrame where `date` is `datetime64[ns]` (no timezone). Verify `ValueError` is raised mentioning timezone.
4. **Valid schema passes without error.** Pass a correctly formatted DataFrame. Verify no exception is raised.

### Negative price detection

5. **Negative close price is flagged.** Insert a row with `close = -5.0`. Verify it appears in `CleaningReport.negative_price_flags` as `(symbol, date_str)`.
6. **Zero open price is flagged.** Insert a row with `open = 0.0`. Verify it appears in `negative_price_flags`.
7. **Negative price rows are NOT dropped.** Verify the row with negative price is still present in the output DataFrame.

### OHLCV consistency

8. **Row where high < low is flagged.** Insert a row with `high = 100.0` and `low = 200.0`. Verify it appears in `consistency_flags`.
9. **Consistency-flagged rows are NOT dropped.** Verify the row is still present in the output.

### Missing value handling

10. **Forward-fill of close works within a symbol group.** Create a DataFrame with symbol "A" having rows [day1: close=100, day2: close=NaN, day3: close=110]. Verify day2 close becomes 100.0 after cleaning. Verify `missing_close_filled == 1`.
11. **Forward-fill does NOT bleed across symbols.** Create two symbol groups: "A" has [day1: close=100, day2: close=NaN] and "B" has [day1: close=NaN, day2: close=200]. Verify "A" day2 is filled with 100. Verify "B" day1 remains NaN (no data before it to fill from) or is handled correctly per pandas ffill semantics (stays NaN because there is nothing to forward-fill from).
12. **Missing open/high/low filled with close value.** Create a row where close is 100.0 but open, high, low are NaN. After cleaning, verify open=high=low=100.0. Verify `missing_ohlv_filled >= 3`.
13. **Missing volume filled with 0.** Create a row with volume=NaN. After cleaning, verify volume=0.0.

### Duplicate date removal

14. **Duplicate dates within symbol: last occurrence kept.** Create two rows for symbol "A" on the same date with different close values (first: 100, second: 110). Verify only the row with close=110 survives.
15. **Duplicate count recorded in CleaningReport.** Verify `duplicates_removed == 1` for the above case.
16. **Duplicates across different symbols are NOT removed.** Two symbols both having data for the same date is normal -- verify both are kept.

### Price floor

17. **Close below price floor is flagged.** Insert a row with `close = 0.50`. Verify it appears in `price_floor_flags`.
18. **Custom price floor is respected.** Call `clean_ohlcv(df, price_floor=50.0)`. Verify a row with `close = 30.0` appears in `price_floor_flags`.

### Output contract

19. **Output columns identical to input columns.** Verify `list(output_df.columns) == list(input_df.columns)`.
20. **Output dtypes identical to input dtypes.** Verify `output_df.dtypes` matches `input_df.dtypes` for all columns.
21. **Output sort order preserved.** Verify output is sorted by `(symbol, date)` ascending.
22. **rows_output == rows_input - duplicates_removed.** Verify this arithmetic identity holds in the CleaningReport.

### CleaningReport

23. **CleaningReport is frozen.** Attempt to assign a field after construction. Verify `FrozenInstanceError` (or `dataclasses.FrozenInstanceError`) is raised.
24. **cleaned_at_ist contains IST timezone offset.** Verify the string contains `+05:30`.
25. **symbols_processed contains all input symbols.** Pass data for 3 symbols. Verify `len(report.symbols_processed) == 3`.

### Input immutability

26. **Input DataFrame is not mutated.** Store a copy of the input before calling `clean_ohlcv`. Verify the original is unchanged after the call.

### Code quality

27. **mypy passes with `--ignore-missing-imports`.** Run mypy on the file.
28. **ruff check passes.** Run ruff check on the file.
29. **No bare except clauses.** Grep the source file for bare `except:` -- verify zero matches.

## 13. Explicit Non-Goals

This module does NOT:

- Fetch data from any external source (yfinance, jugaad-data, Screener.in, NSE). It only cleans DataFrames handed to it.
- Write to SQLite or any database. Logging goes to stdout only (via Python's `logging` module). Database logging is `src/utils/logger.py`'s job (Phase 1, step 6).
- Send notifications via Telegram or Gmail.
- Apply strategy filters. The price floor of 1.0 INR is a data sanity check (detecting corrupt data). The strategy's price filter (> 50 INR) lives in `src/strategy/quality_filter.py`.
- Validate data quality scores. That is `src/data/validator.py`'s role. The cleaner repairs; the validator judges.
- Drop anomalous rows. It flags them and leaves them for downstream consumers to decide.
- Backfill gaps. It forward-fills NaN values within existing rows but does not insert new rows for missing trading days (gap detection is the validator's job).
- Change the DataFrame schema in any way (no column additions, removals, or dtype changes).
- Handle fundamentals data (ROE, D/E, EPS). It operates on OHLCV DataFrames only.

## 15. Pipeline Integration

The cleaner's position in the Phase 1 pipeline:

```
fetcher.fetch_ohlcv()  -->  cleaner.clean_ohlcv()  -->  validator.validate_data()
        |                         |                           |
   Raw OHLCV DataFrame     Cleaned OHLCV DataFrame     DataQualityReport
   (Section 5.1 contract)  (same contract, repaired)   (scores + flags)
```

The `main.py` pipeline (Phase 1, step 9) will call:

```python
raw_df = fetch_ohlcv(symbols, start_date, end_date)
cleaned_df, cleaning_report = clean_ohlcv(raw_df)
quality_report = validate_data(cleaned_df, fundamentals_df, db_path)
```

The `CleaningReport` is available for logging alongside the `DataQualityReport` but is not consumed by the validator. They are independent audit trails.
